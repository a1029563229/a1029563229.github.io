Vue.js 服务端渲染指南

注意: 本篇指南要求Vue和相关依赖库的最低版本为:

  • vue & vue-server-renderer >= 2.3.0
  • vue-router >= 2.5.0
  • vue-loader >= 12.0.0 & vue-style-loader >= 3.0.0

如果你之前使用vue2.2做服务端渲染, 你会发现有一些代码的结构和现在的有一点 差异 (新版的 runInNewContext选项应设为 false). 你现有的项目可以继续运行, 但我们建议迁移到最新版本.

什么是服务端渲染 (SSR)?

Vue.js是用来构建客户端应用的一个框架. 默认情况下，Vue的组件操作和生产dom，然后输出给浏览器. 然而，它也可以读取相同的组件插入到服务器的Html字符串中,然后直接发送给浏览器, 最后我们的’混合应用’变成一个完全交互式的应用程序展现在客户端.

服务端vue应用也可以认为是“同构”或者“通用”, 从这个意义上来说，你的一套代码可以同时在客户端和服务端运行。

为什么使用SSR?

相比较传统的SPA应用 (单页面应用), SSR的主要优势在于:

  • 更好的 SEO, 搜索引擎的爬虫会抓取完全渲染的页面.

    直到现在, Google和Bing能够很好的同步索引Javascript应用. 同步是个关键词. 如果你的应用有一个加载页, 然后通过Ajax来获取主要内容, 爬虫不会等待他们完成. 这就意味着如果你要在页面异步获取内容的情况做SEO优化的话, SSR是至关重要的.

  • 更快的加载速度, 尤其是在网速慢或者设备不好的情况下. 服务端渲染不需要等待所有js下载并且执行就能直接展示页面, 所以你的用户能直接看到一个完全渲染好的页面，这样会有更好的用户体验.对于时间转换为效率的应用程序来说，服务端渲染更为重要.

考虑使用SSR时也有一些需要权衡的因素:

  • 开发的限制. 有一些浏览器特定代码只能在某些生命周期钩子才能使用; 有一些外部依赖库需要特别处理之后才能在服务端应用程序上运行.

  • 涉及到构建和部署要求.普通的SPA应用可以部署在任何服务器，但是SSR应用只能运行在Node的环境.

  • 更大的服务器荷载. 在Node上，比起加载几个静态文件，加载一个完整的应用程序需要更多的CPU占用率, 所以，如果你希望有更快的速度, 需要准备相应的服务器负载和明确的使用缓存策略。

给你的应用使用SSR之前, 首先你需要想清楚你是否真的需要它. 这主要取决于你的应用是否很需要“时间转化为效率”. 举个栗子, 如果你只是制作一个内网应用，几百毫秒的加载速度提升并没有什么作用， 使用SSR纯属多此一举. 但是, 加载速度对于你来说非常重要的话, SSR能帮你实现最佳的初始负载性能.

SSR vs 预渲染（Prerendering）

如果你只是用服务端渲染来改善一个少数的营销页面（如 首页，关于，联系 等等）的SEO，那你可以用预渲染替换。预渲染不像服务器渲染那样即时编译HTML,预渲染只是在构建时为了特定的路由生成特定的几个静态页面。其优势是预渲染的设置更加简单，可以保持前端是一个完整的静态站。

你用webpack可以很简单地通过prerender-spa-plugin来添加预渲染，它被广泛地用在Vue应用上 - 事实上，创建者也是Vue核心团队成员之一。

关于本篇文章

本篇文章重点是讲解如何在Node服务器上运行SSR SPA应用. 混合与其他后台设置Vue SSR是一个自己的主题，不包括在本指南。

本篇文章会比较深入，假设你已经熟悉Vue, 并且掌握一些关于Node和Webpack的知识. 如果你想要一个更高级的解决方案, 你可以去试试 Nuxt.js. 它和Vue的栈是差不多的，但是有更多的样板, 并且提供一些额外的功能，例如生成静态站点. 然而，如果你需要控制你的项目结构，它可能不太适用，总而言之，本篇文章能让你更好的理解它的工作原理。

如果你一直读下去的话, 你可以结合这个 HackerNews Demo来学习,这个demo涵盖了本篇文章的大部分知识点.

最后, 本篇文章的解决方案也不是明确的 - 我们可以通过它来更好的工作, 但是我们会优化他的.文档会在以后继续更新 - 并且你可以参与其中!
